## 1. 서론

### 1.1 문서의 목적 및 범위

본 프로젝트 관리 계획서는 **에어 드라이브(Air Drive)** — 경량화된 팀 협업을 위한 분산 파일 관리 및 히스토리 추적 시스템 — 의 전반적인 개발 계획을 정리하여, 프로젝트를 진행하는 데 있어 각 세부 사항의 지표로 삼는 것을 목적으로 한다.

본 문서는 다음과 같은 범위의 지표로 적용된다.

- ① **프로젝트 개요**: 프로젝트 정의, 주요 기능, 기존 유사 소프트웨어 분석
- ② **개발 계획 및 환경**: 개발 절차 모형, 내용 및 일정, 소프트웨어/하드웨어 환경
- ③ **팀 구성**: 팀 구조, 역할 및 책임
- ④ **품질 관리**: 팀 미팅, 변경 사항 및 산출물 관리
- ⑤ **산출물**: 산출물 정의, 작성일 및 담당자

---

### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의

본 시스템은 대규모 클라우드 서비스의 복잡함을 걷어내고, **팀 단위의 빠른 자료 공유에 최적화된 경량 웹 스토리지**를 제공한다.

사용자는 웹을 통해 파일을 업로드하고, 팀원 간의 실시간 댓글 피드백을 통해 문서의 수정 방향을 공유할 수 있다. 또한 변경된 부분만 관리하는 **버전 제어 기능**을 제공하여 팀의 데이터 자산을 안전하게 보호한다.

#### 1.2.2 핵심 기능

| ID | 기능명 | 설명 |
|----|--------|------|
| F01 | 드래그 앤 드롭 파일 관리 | 복잡한 설치 없이 브라우저에서 파일을 바로 올리고 폴더별로 분류하는 직관적 관리 기능 |
| F02 | 시간 제한부 공유 링크 생성 | 보안을 위해 특정 시간(예: 1시간, 24시간) 동안만 활성화되는 임시 다운로드 링크 발급 기능 |
| F03 | 파일별 협업 댓글 | 파일 자체에 댓글을 달아 수정 이력이나 팀원 간의 의견을 기록하고 보존하는 기능 |
| F04 | 최근 활동 로그 추적 | 누가 어떤 파일을 언제 수정하거나 다운로드했는지 실시간으로 보여주는 가시성 제공 기능 |
| F05 | 키워드 기반 고속 필터링 | 파일명, 확장자, 태그를 조합하여 수천 개의 파일 중 필요한 자료를 즉시 찾아내는 검색 기능 |

#### 1.2.3 기존 유사 소프트웨어 조사 및 분석

| 서비스 | 장점 | 한계 |
|--------|------|------|
| SendAnywhere | 1회성 전송 속도가 빠름 | 파일 버전 관리나 팀 단위의 지속적인 보관 및 협업 불가 |
| MS OneDrive | 기능이 강력함 | 설정이 매우 복잡하고 시스템 리소스 소모가 커 소규모 팀이 가볍게 쓰기엔 진입 장벽이 높음 |
| **Air Drive** | 경량화·팀 협업 특화 | - |

---

### 1.3 프로젝트 목표

- 설치 없이 브라우저만으로 팀 파일을 즉시 공유하고 관리할 수 있는 경량 웹 스토리지 시스템 구축
- 버전 제어 및 활동 로그를 통해 팀의 협업 히스토리를 투명하게 추적
- SendAnywhere의 간편함과 OneDrive의 협업 기능을 결합한 중간 지점의 솔루션 제공

---

### 1.4 용어 정의

| 용어 | 설명 |
|------|------|
| 버전 제어 | 파일의 변경 이력을 관리하고, 이전 상태로 복원할 수 있는 기능 |
| 시간 제한부 링크 | 일정 시간이 지나면 자동으로 만료되는 임시 공유 URL |
| 활동 로그 | 파일의 업로드, 수정, 다운로드 등 사용자 행동 기록 |
| 드래그 앤 드롭 | 마우스로 파일을 끌어다 놓는 방식의 UI 인터랙션 |

## 2. 개발 계획

### 2.1 개발 절차 모형

요구사항이 점진적으로 구체화될 것으로 예상되고, 팀원 간 빠른 피드백 반영이 중요하기 때문에 **Agile 방법론 중 SCRUM 모델**을 채택한다.

설계 및 분석 방법론으로는 컴포넌트 단위의 분리와 재사용이 용이한 **컴포넌트 기반 설계 방식**을 적용한다.

---

### 2.2 WBS (Work Breakdown Structure)

```
Air Drive 시스템 개발
├── 요구사항 분석
│   ├── 계획서 검토
│   ├── 유사 서비스 조사 (SendAnywhere, OneDrive 등)
│   ├── 요구사항 분석 및 정의
│   └── 핵심 기능 정의 (F01~F05)
├── 설계
│   ├── 분석서 검토
│   ├── 시스템 구조 설계 (파일 저장, 버전 관리)
│   └── 상세 설계 (UI/UX, API, DB 스키마)
├── 개발
│   ├── 기능 구현 (F01~F05)
│   ├── 서버 및 DB 구축
│   └── 각 기능 통합
└── 테스트
    ├── 단위 테스트
    ├── 통합 테스트
    └── 발표 자료 작성
```

## 4. 품질 관리

### 4.1 팀 미팅 계획

**(1) Sprint Planning Meeting**
- 팀 미팅 시간: 매주 월요일
- 토의 내용: Sprint 목표와 Sprint Backlog 계획
- 참석자: 팀원 전체, 팀장

**(2) Daily Scrum Meeting**
- 팀 미팅 시간: 매일 진행 (비동기 가능, Slack/Discord 활용)
- 토의 내용: 어제 한 일, 오늘 할 일, 장애 현상 공유
- 참석자: 팀원 전체, 팀장

**(3) Sprint Review**
- 팀 미팅 시간: Sprint 종료 시점
- 토의 내용: Sprint 기간 동안 팀원 전체가 각자 한 일 리뷰
- 참석자: 팀원 전체, 팀장

**(4) Sprint Retrospective**
- 팀 미팅 시간: Sprint 종료 시점
- 토의 내용: 개선할 것, 잘한 것, 새로 시작할 것 도출
- 참석자: 팀원 전체, 팀장

**(5) 비정기 회의**
- 프로젝트 진행에 문제가 발생할 경우 팀장 또는 지도교수 요청에 의해 소집

---

### 4.2 변경 사항 관리 방법

- 기능 변경은 팀원 전체의 합의(다수결)가 필요하며, 의견 충돌 시 팀장이 최종 결정권을 가진다.
- 변경 사항 발생 시 전·후에 대해 반드시 문서로 기록한다.
- 관련 산출물은 팀원 전원이 공유 및 보관한다.
- 주요 변경 결정 사항은 지도교수님께 보고한다.
